## Objective

Implement the EKS cluster module per DESIGN-0002. The module today is a scaffold:
`variables.tf`, `locals.tf`, and `versions.tf` are populated, `main.tf` is a
single comment header, and `outputs.tf` exports five workload-tier IAM role
ARNs whose backing resources don't exist. This IMPL turns the scaffold into a
working module that downstream consumers (managed-node-group, addons,
pod-identity-access) can read via `terraform_remote_state`.

**Implements:** DESIGN-0002 — EKS Cluster Module, with the workload-IAM scope
deliberately removed (see [Supersedes](#supersedes)).

**Constrained by:** ADR-0001 (remote-state composition), ADR-0002 (Pod Identity
trust), ADR-0003 (no addons here), ADR-0011 (AWS-API-only — no `kubernetes`
provider).

## Scope

### In Scope

- `aws_eks_cluster.this` with envelope encryption, `endpoint_public_access`
  enabled (for `kubectl` from outside the VPC), control-plane logging.
- Cluster IAM role with `AmazonEKSClusterPolicy` — this is the only IAM role
  remaining in the cluster module. It is control-plane, not workload IAM.
- Optional module-managed KMS key (`var.kms_key_arn == null`).
- Shared node security group exported as `node_security_group_id`.
- SSO Access Entry + policy association (gated by `sso_access_enabled`).
- **Hoist** `data.aws_region`, `data.aws_iam_account_alias`, `data.aws_vpc`,
  and `data.aws_subnets.*` to Boilerplate-generated Terragrunt inputs
  (`var.tags`, `var.region`, `var.remote_state_bucket`, `var.vpc_name`) per
  ADR-0001.
- **Delete** the five workload-tier IAM role outputs currently in
  `modules/eks/cluster/outputs.tf` (`cluster-autoscaler_arn`,
  `pod_cw_metrics_arn`, `pod_fluentd_logs_arn`, `alb_role_arn`,
  `external_dns_arn`). They will be re-homed under DESIGN-0004
  (`pod-identity-access`).
- libtftest suite covering plan-time invariants per DESIGN-0002 §Testing.
- `USAGE.md` regenerated by terraform-docs.

### Out of Scope

- All **workload-tier IAM** (cluster-autoscaler, ALB controller,
  external-dns, FluentD, CloudWatch metrics) — moves to DESIGN-0004 so each
  role lives next to the workload that uses it.
- Pod Identity Associations — DESIGN-0003 (addon-managed) + DESIGN-0004
  (workload-scoped).
- Managed addons (Pod Identity Agent, VPC CNI, kube-proxy, CoreDNS,
  EBS/EFS CSI) — DESIGN-0003 per ADR-0003.
- Any `kubernetes`, `kubectl`, or `helm` provider usage — ADR-0011.
- Provisioning the S3 backend, the VPC, or the SSO permission sets.

## Supersedes

This IMPL deliberately narrows DESIGN-0002 in one place:

- **DESIGN-0002 §"Controller IAM roles"** — the five Pod-Identity-trusting
  controller roles (`cluster-autoscaler`, ALB controller, external-dns,
  FluentD, CW metrics) are **not** implemented in this module. They re-home
  to DESIGN-0004 (`pod-identity-access`), keeping each role next to the
  workload that consumes it.
- **Resolves DESIGN-0002 §"Open Questions / Still open" #3** in favor of
  full delegation to DESIGN-0004.

DESIGN-0002 and DESIGN-0004 should be amended to reflect this rebalancing —
tracked in [Follow-ups](#follow-ups).
