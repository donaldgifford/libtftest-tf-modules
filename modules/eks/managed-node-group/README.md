# managed-node-group

Secure EKS managed node group with gVisor sandboxing, minimal node IAM, IMDSv2 +
hop-limit 2, and ARM64/Graviton as the default architecture.

- Terraform interface: [USAGE.md](./USAGE.md) (generated by `terraform-docs`)
- Design:
  [`docs/design/0001-secure-eks-managed-node-group-with-gvisor.md`](../../../docs/design/0001-secure-eks-managed-node-group-with-gvisor.md)
- Decisions: ADR-0001, ADR-0002, ADR-0005, ADR-0006, ADR-0007, ADR-0008,
  ADR-0009, ADR-0010, ADR-0011, ADR-0012

## Out-of-band: the gVisor `RuntimeClass`

This module **does not** create the gVisor `RuntimeClass`. Per
[ADR-0011](../../../docs/adr/0011-runtimeclass-delivered-out-of-band-not-by-terraform.md),
Terraform modules in this repo manage AWS API resources only; Kubernetes-API
objects are delivered out-of-band.

Workloads in this node group opt into gVisor by setting
`spec.runtimeClassName: gvisor`. That reference resolves only if a
cluster-scoped `RuntimeClass` named `gvisor` exists. The manifest is the same in
every cluster тАФ one per cluster, regardless of how many secure node-group
instantiations (`arm64` + `amd64`) the cluster runs.

### The manifest

```yaml
# gvisor-runtimeclass.yaml
apiVersion: node.k8s.io/v1
kind: RuntimeClass
metadata:
  name: gvisor
handler: runsc
scheduling:
  nodeSelector:
    workload-class: secure
    runtime: gvisor
  tolerations:
    - key: workload-class
      value: secure
      operator: Equal
      effect: NoSchedule
```

The values are stable constants matching what this module applies:

| Field                     | Value                                     | Where it comes from                                             |
| ------------------------- | ----------------------------------------- | --------------------------------------------------------------- |
| `handler`                 | `runsc`                                   | containerd handler registered by user data (ADR-0005, ADR-0008) |
| `scheduling.nodeSelector` | `workload-class=secure`, `runtime=gvisor` | `aws_eks_node_group.labels`                                     |
| `scheduling.tolerations`  | `workload-class=secure:NoSchedule`        | `aws_eks_node_group.taint`                                      |

### Option A тАФ Apply via `kubectl`

For homelab / dev clusters and environments without a GitOps tool:

```sh
aws eks update-kubeconfig --region "$AWS_REGION" --name "$CLUSTER_NAME"
kubectl apply -f gvisor-runtimeclass.yaml
```

Verify:

```sh
kubectl get runtimeclass gvisor -o yaml
```

Run this once per cluster after the cluster + node-group Terraform applies have
completed and at least one secure node is `Ready`.

### Option B тАФ Apply via Argo CD + Kustomize

Recommended for production. The manifest lives in a GitOps repository under a
per-cluster overlay; Argo syncs it like every other cluster-scoped object.

Layout (consumer's GitOps repo):

```
clusters/<env>/<cluster-name>/
тФЬтФАтФА base/
тФВ   тФФтФАтФА gvisor-runtimeclass/
тФВ       тФЬтФАтФА kustomization.yaml
тФВ       тФФтФАтФА runtimeclass.yaml
тФФтФАтФА overlays/
    тФФтФАтФА <cluster-name>/
        тФФтФАтФА kustomization.yaml
```

**`base/gvisor-runtimeclass/runtimeclass.yaml`** тАФ the manifest above, verbatim.

**`base/gvisor-runtimeclass/kustomization.yaml`**:

```yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
resources:
  - runtimeclass.yaml
```

**`overlays/<cluster-name>/kustomization.yaml`**:

```yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
resources:
  - ../../base/gvisor-runtimeclass
```

**Argo `Application`** (in the Argo control-plane cluster):

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: <cluster-name>-gvisor-runtimeclass
  namespace: argocd
spec:
  project: cluster-bootstrap
  source:
    repoURL: https://github.com/<org>/<gitops-repo>.git
    targetRevision: main
    path: clusters/<env>/<cluster-name>/overlays/<cluster-name>
  destination:
    server: https://<cluster-api-endpoint>
    # namespace omitted тАФ RuntimeClass is cluster-scoped
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    syncOptions:
      - CreateNamespace=false
      - ServerSideApply=true
```

Add the `gvisor-runtimeclass` Application to the same Argo sync wave as other
cluster-bootstrap manifests (NetworkPolicies, admission webhooks, etc.). It
should sync after the node-group Terraform apply has completed.

## Workload opt-in (consumer reference)

Once the `RuntimeClass` exists, workloads opt in:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: secure-app
spec:
  template:
    spec:
      runtimeClassName: gvisor
      # nodeSelector + toleration are injected by RuntimeClass.scheduling
      # Add kubernetes.io/arch only if the workload targets a specific arch
      # and its container image is single-arch:
      # nodeSelector:
      #   kubernetes.io/arch: arm64
      containers:
        - name: app
          image: <multi-arch-image>:<tag>
```

Multi-arch container images schedule on either architecture in mixed-arch
clusters without an arch nodeSelector.
